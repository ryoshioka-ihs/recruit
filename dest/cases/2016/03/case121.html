SCCM 更改

背景

今回のお客様は、既存System Center 2007 Configuration Manager環境から新規にSystem Center 2012 R2 Configuration Manager への更改のための、システム構成、構築を検討しておりました。

本件では、System Center 2012 R2 Configuration Manager の管理機能であるプライマリサイトの構築。

また、各拠点に、System Center Configuration Manager 機能のセカンダリサイトサーバー構築の対応をさせていただきました。


弊社にて、System Center 2012 R2 Configuration Managerの管理機能であるプライマリサイトおよび、セカンダリサイトサーバーの構築を対応させていただきました。

システム構成図

作業効果

セカンダリサイトサーバーによる帯域の分散

お客様環境では、将来クライアント端末の増加を予定されておりました。


クライアント端末増加に伴い、セカンダリサイトサーバーの構築を行いました。
セカンダリサイトサーバーを構成することにより、プロキシ管理ポイントとして作動させることが可能になります。 通常プロキシ管理ポイントを構成しない場合、クライアントはプライマリサイトと直接通信を行う動作をし、境界内にあるクライアント数が増加するとネットワーク帯域幅も増加します。


しかし、プロキシ管理ポイントを構成することにより、クライアントの情報は、プロキシ管理ポイントに寄って圧縮さセカンダリサイトからプライマリサイトサーバーに送信されます。


これにより、クライアント端末増加を考慮した、構成になっております。

弊社利用による効果

SCCM2007 クライアントエージェントの切り換え

お客様環境では、SCCMのシステム更改に伴い、既存SCCM環境に管理されたクライアント端末を移行する必要がありました。本件では、エージェントの移行を実施するため、スクリプトの作成を行いました。


SCCmエージェントの移行には、二種類の方法があります。
一つ目は、SCCM2012の機能を使用し自動的に移行を行う方法。
二つ目は、既存のSCCM2007にてパッケージを作成し、移行用のスクリプトを配布し移行を行う方法があります。


前者は、設定を行えば自動的に行われますが、管理クライアントの台数を考えるとネットワークのトラフィックが増加し、既存のネットワークに影響を与えてしましいます。
これにより、今回はパッケージ配布による移行を実行いたしました。
こちらのメリットは、クライアントの意向タイミングを図れることにあります。
移行するクライアントの対象をし絞り、ネットワークのトラフィックの増加を考慮する必要がなくなります。
